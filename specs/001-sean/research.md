# Research: WCAG-Compliant Document-to-HTML Converter

**Phase 0 Output** | **Date**: 2026-03-11

## R1: WCAG 2.1 AA Automated Validation

**Decision**: Use axe-core for automated WCAG validation on all generated
HTML output and the web UI itself.

**Rationale**: axe-core is the industry standard for automated accessibility
testing. It catches ~57% of WCAG issues automatically (the remainder require
manual testing). It integrates with pytest (via axe-selenium), Jest (via
jest-axe), and Playwright (via @axe-core/playwright). The NC Digital
Accessibility and Usability Standard v1.1 references automated testing tools.

**Alternatives considered**:
- WAVE API: Good but requires external API calls; axe-core runs locally
- pa11y: Strong CLI tool but less ecosystem integration than axe-core
- Lighthouse: Broader scope; axe-core is more focused on accessibility

## R2: DOCX Extraction Approach

**Decision**: Use python-docx for Word document extraction.

**Rationale**: python-docx provides direct access to Word's document.xml
structure including styles (Heading 1-6), paragraphs, tables, images, and
lists. It preserves the semantic structure that Word authors create, making
it the richest source of heading/list hierarchy for accessible HTML.

**Alternatives considered**:
- mammoth.js: Good HTML output but JavaScript-only; adds complexity to
  Python backend
- LibreOffice CLI conversion: Heavyweight; requires system dependency;
  output HTML is not semantically clean
- Azure Document Intelligence for DOCX: Overkill; DOCX already has
  structured content

## R3: PPTX Extraction Approach

**Decision**: Use python-pptx for PowerPoint extraction.

**Rationale**: python-pptx provides access to slide content, shapes, text
frames, tables, images, and speaker notes. Each slide maps naturally to an
HTML `<section>` with the slide title as a heading.

**Alternatives considered**:
- LibreOffice CLI: Same drawbacks as for DOCX
- Azure Document Intelligence: Does not support PPTX directly

## R4: Frontend Framework

**Decision**: React 18 with Next.js 14 (App Router) on Bootstrap 5.

**Rationale**: Next.js provides SSR for fast initial load, API routes for
backend communication, and static export capability for Azure Static Web
Apps deployment. Bootstrap 5 is the CSS framework used by the NCDIT Digital
Commons design system (NC.gov sites use `bootstrap@5.1.3`). React component
architecture enables reusable accessible components.

**Alternatives considered**:
- Plain HTML/JS: Simpler but lacks component reuse, state management for
  progress tracking, and SSR
- Vue/Nuxt: Viable but React has larger ecosystem for accessibility tooling
  (jest-axe, @axe-core/playwright)
- Angular: Heavier; overkill for this application scope

## R5: Real-Time Progress Updates

**Decision**: Polling with SAS-token-scoped blob metadata queries.

**Rationale**: Azure Functions consumption plan doesn't support WebSockets.
Polling blob metadata (processing status stored as blob metadata tags) every
3-5 seconds provides near-real-time updates without additional infrastructure.
Azure Blob Storage already stores the documents.

**Alternatives considered**:
- Azure SignalR Service: Real-time but adds cost and another service
- Server-Sent Events (SSE): Not supported by Azure Functions consumption
  plan
- Azure Queue + polling: More complex; blob metadata is simpler

## R6: File Upload to Azure Blob Storage

**Decision**: Direct browser-to-blob upload using SAS tokens generated by
an Azure Functions HTTP endpoint.

**Rationale**: Bypasses the 100MB Azure Functions request limit. The frontend
requests a short-lived SAS token from an HTTP-triggered function, then
uploads directly to blob storage using the Azure Storage Blob JavaScript SDK.
This enables progress tracking on large files and scales without function
timeout issues.

**Alternatives considered**:
- Upload through Azure Functions: Limited by request body size and timeout
- Azure Front Door with upload: Adds complexity; SAS tokens are simpler
- Managed upload widget: No NC.gov-branded option exists

## R7: NCDIT Digital Commons Implementation

**Decision**: Custom CSS design tokens file mapping Digital Commons colors,
typography, and spacing to CSS custom properties, used with Bootstrap 5
utility classes and the frontend-design skill for polish.

**Rationale**: Digital Commons doesn't publish a standalone npm package or
React component library. The approach is to extract the design tokens (colors,
fonts, spacing) from the NC.gov reference implementation and apply them via
CSS custom properties layered on Bootstrap 5. The frontend-design skill adds
production-grade aesthetics within those brand constraints.

**Alternatives considered**:
- Drupal theme port: NC.gov runs Drupal; porting the full theme is overkill
- Screenshot-matching: Fragile; design tokens are more maintainable

## R8: HTML Output Self-Containment

**Decision**: All converted HTML includes inline CSS and base64-encoded images
(with external image option for large documents).

**Rationale**: Self-contained HTML can be published directly on state websites
without managing asset dependencies. For documents with many images, an option
to reference external image files (stored alongside the HTML) prevents
excessive file sizes.

**Alternatives considered**:
- External CSS: Requires hosting; breaks self-containment
- PDF.js viewer: Not accessible; defeats the purpose of HTML conversion
